# 📜 RULE REGISTRY — Data Engineering & Analytics

34 data-specific decision rules, best practices, and anti-patterns.

| # | Rule | Category | Severity | Anti-Pattern (DO NOT) |
|---|---|---|---|---|
| R001 | Every dbt model must have a `unique` test on its primary key and `not_null` tests on all required columns — uniqueness violations in core models corrupt every downstream analysis; dbt test failures block CI pipeline merge | Data Quality | 🟥 CRITICAL | `dim_customers` with no unique test on `customer_id` — duplicate customers inflate revenue in every dashboard |
| R002 | Never use `SELECT *` in production data models — always enumerate columns explicitly; schema changes in source tables silently change downstream model structure | Data Modeling | 🟥 CRITICAL | `SELECT * FROM raw_orders` → source adds column → downstream model schema changes → dashboard breaks |
| R003 | Incremental models must use `unique_key` and handle late-arriving facts — without unique_key, re-running duplicates data; without late arrival handling, corrected source data never updates target | Data Modeling | 🟥 CRITICAL | `is_incremental()` filter on `created_at > max(created_at)` — corrected orders from 3 days ago never picked up |
| R004 | Pipeline tasks/DAGs must be idempotent — running the pipeline twice with the same input produces the same output; use MERGE/UPSERT or DELETE+INSERT in a transaction; never INSERT INTO without deduplication | Pipeline Reliability | 🟥 CRITICAL | Airflow task does `INSERT INTO daily_sales SELECT * FROM orders WHERE date='{{ ds }}'` — manually re-running duplicates all rows |
| R005 | Data quality checks must run at every transformation stage, not just the end — shift-left: validate data entering each stage before transformation; catch errors at the earliest possible point | Data Quality | 🟧 HIGH | 2-hour pipeline fails at final MERGE because source column type changed — all 2 hours wasted |
| R006 | All PII/PHI/PCI must be classified, masked in non-production, and access-controlled — dynamic data masking, policy tags, or column-level security; no raw PII in dev/staging; audit access to PII tables | Data Governance | 🟥 CRITICAL | Customer email/phone/ssn in development tables with no masking — intern's laptop stolen |
| R007 | Partitioning strategy must match query patterns — partition by most common filter (usually date); cluster by high-cardinality WHERE/JOIN columns; never partition by high-cardinality columns (user_id) — millions of tiny partitions kill performance | Query Performance | 🟧 HIGH | BigQuery table partitioned by user_id (50M users) → 50M partitions → metadata scan overhead kills performance |
| R008 | Data freshness must be monitored with SLAs — every source table has freshness SLA (warn_after, error_after); freshness failures alert the owning team; stale data in dashboards erodes trust | Data Quality | 🟧 HIGH | "Real-time dashboard" showing 3-day-old data because ingestion pipeline silently failed |
| R009 | Data lineage must be complete and automated — every transformation step recorded (dbt docs, OpenLineage, DataHub); understand upstream dependencies before changing a model; never manually update data without recording lineage | Data Governance | 🟧 HIGH | "What tables does the revenue dashboard depend on?" → original builder left the company |
| R010 | Never hardcode credentials in pipeline code or config files — use Airflow Connections/Secrets Backend, Prefect Blocks, or env vars from secrets manager; rotate service account keys every 90 days | Security | 🟥 CRITICAL | `snowflake.connector.connect(user='ANALYST', password='Summer2024!')` hardcoded in profiles.yml |
| R011 | JOIN keys must be of the same data type — implicit type coercion prevents index usage, produces incorrect results, and is silently destructive; cast explicitly if types differ and document why | Query Performance | 🟧 HIGH | `o.customer_id` is INT, `c.customer_id` is VARCHAR → 45-minute query instead of 30 seconds |
| R012 | Backfill procedures must be documented and tested — every pipeline must have a runbook for backfilling a date range; backfill must be idempotent; test backfill in staging; have a rollback plan | Pipeline Reliability | 🟧 HIGH | "Can we backfill Q2?" → pipeline was built 2 years ago, no one knows how |
| R013 | Schema evolution must be managed with compatibility guarantees — Avro/Protobuf schema registries enforce FULL_TRANSITIVE; add fields with defaults, never remove required fields, never change field types; communicate breaking changes with ≥2 week notice | Data Contracts | 🟧 HIGH | Producer changes `amount` from double to string → all consumer pipelines break on deserialization |
| R014 | Materialization strategy must be intentional — views for staging, ephemeral for intermediate used by 1-2 models, table for core dimensions/facts, incremental for large fact tables; never default all models to table | Data Modeling | 🟧 HIGH | 500 dbt models all materialized as table — 200GB near-identical intermediates, $15K/month bill, 8-hour runs |
| R015 | Data retention policies must align with legal requirements AND query patterns — never keep raw data forever "just in case" (GDPR/CCPA deletion right, storage cost); define retention in code; aggregate data may have longer retention | Data Governance | 🟧 HIGH | Raw clickstream from 2015 at full granularity — 5PB, $500K/year, contains PII |
| R016 | Monitors and alerts must be defined for pipeline health — pipeline failure→P1, quality ERROR→P2, quality WARN→P3, pipeline running longer than SLA→P3; no silent failures | Observability | 🟧 HIGH | Critical financial pipeline fails nightly — first noticed by CFO during board meeting with 0 revenue |
| R017 | Dashboard queries must be optimized for user-facing latency — BI dashboards <5s; pre-aggregate in materialized views if source query >10s; use extracts for frequently accessed dashboards; never raw table scan on every load | Query Performance | 🟧 HIGH | Executive dashboard scanning 50B rows on every page load — 45-second load time |
| R018 | Documentation must be written for data consumers, not producers — model descriptions explain WHAT data represents (business definition), not HOW it was built; column descriptions give examples; catalog searchable by business terms | Data Governance | 🟨 MEDIUM | Model description: "Joins orders CTE with customers CTE using LEFT JOIN on customer_id" — business analyst has no idea |
| R019 | Test data in development must never contain production PII — use synthetic data, sampled data with PII columns hashed/masked, or data generators; never restore full production datasets to developer laptops | Data Security | 🟥 CRITICAL | `pg_dump production | pg_restore development` — production PII on 15 developer laptops |
| R020 | Feature engineering pipelines must prevent training/serving skew — features computed identically in offline and online paths; use same code/library; validate offline and online feature values match for same entity at same timestamp | ML Infrastructure | 🟧 HIGH | Spark rounding logic vs Go rounding logic → model performs 40% worse in production |
| R021 | SQL formatting must be consistent — leading commas, lowercase keywords, 4-space indent, explicit AS for aliases, CTEs over subqueries; run sqlfluff or dbt-format in CI | Code Quality | 🟨 MEDIUM | 200-line CTE with mixed case, no indentation, aliases `a`, `b`, `c` — bug in JOIN unnoticed for 6 months |
| R022 | Large-scale processing must use appropriate tooling — use Spark/Flink for TB+ scale; use dbt for SQL on warehouses (push compute); use COPY/INSERT with batch size tuning; profile before scaling vertically | Query Performance | 🟧 HIGH | `for row in cursor.fetchall(): insert_one(row)` — 100M rows one-at-a-time, 47 hours estimated |
| R023 | Streaming pipelines must handle late-arriving data with watermarking — define watermark threshold; data later than watermark is discarded (with metric) or routed to dead-letter topic; never hold state indefinitely | Stream Processing | 🟧 HIGH | Streaming JOIN with no watermark — state store grows unbounded, OOMs Flink task manager |
| R024 | All data models must have an owner and a purpose — owner team, business purpose, freshness SLA, sensitivity classification; orphan tables flagged for archival; prevent data swamp | Data Governance | 🟨 MEDIUM | 10,000 tables, 3,000 never queried, 2,000 have no owner — $200K/year in storage |
| R025 | Cross-database queries must be minimized — prefer ETL/ELT to bring data into a single warehouse; if cross-DB unavoidable, use materialized external tables or scheduled sync with freshness monitoring | Architecture | 🟧 HIGH | Dashboard querying Snowflake, PostgreSQL, MongoDB in real-time — 30-second load, 3 failure points |
| R026 | Data contracts must be enforced at ingestion — define schema expectations at point of landing; reject or quarantine non-compliant data; never silently accept malformed data that corrupts downstream | Data Contracts | 🟧 HIGH | Kafka topic with Protobuf schema, producer sends JSON → deserialization fails silently → 20% data loss |
| R027 | Metric definitions must be centralized and version-controlled — define in dbt Metrics, Cube, or LookML; never define the same metric differently in 3 dashboards; metric changes are code changes → PR → review → deploy | Analytics | 🟧 HIGH | "Revenue" defined 3 ways: gross (dashboard A), net of refunds (B), net of refunds+chargebacks (C) — 3 different answers |
| R028 | Ad-hoc analysis must be reproducible — explicit dependency versions, seed/sampled data, clear input/output; commit to shared repo; never "works on one person's laptop" | Analytics | 🟨 MEDIUM | `.ipynb` with hardcoded paths to `/Users/jane/Downloads/data_dump_v3_final.csv`, missing deps |
| R029 | Data must be validated against source systems regularly — reconciliation: row counts + key financial totals match within 0.01%; schedule daily or per-load; alert on mismatch; diverging from source is invisible corruption | Data Quality | 🟧 HIGH | Warehouse says 1,000,000 orders, source says 997,432 — no one is checking |
| R030 | Cost-aware data engineering — use EXPLAIN before creating large models; set cost budgets per query/user; prefer columnar formats (Parquet/ORC) over row-based; use table sampling for development; autoscale with max limits | Cost Management | 🟧 HIGH | `SELECT * FROM 50TB-table` in dev to "check schema" → $5,000 BigQuery scan |
| R031 | Historical data must be archived, not left in operational tables — data older than query window moves to cold storage (S3 Glacier, BigQuery long-term, Snowflake iceberg); archived data queryable without restore; never just delete | Data Management | 🟨 MEDIUM | 10 years of raw events in hot warehouse — 99.7% of queries touch last 90 days |
| R032 | Slowly Changing Dimensions (SCD) must use the right type intentionally — Type 1 for correctable errors (typo fix), Type 2 for true historical changes (address, price), Type 3 for single-version comparisons | Data Modeling | 🟧 HIGH | Every dim table is SCD Type 2 — dim_product with 47 versions because typo corrected 46 times |
| R033 | Dead-letter queues must have alerting and replay capability — DLQ depth monitored; messages have failure reason, original timestamp, retry count; replay script available; never silently drop messages | Stream Processing | 🟧 HIGH | DLQ with 15M messages, no alerting because "too noisy" — $4.7M unprocessed transactions |
| R034 | All data pipeline infrastructure must be version-controlled and deployed via CI/CD — dbt, Airflow DAGs, Spark jobs, schema migrations in git; changes via PR with review; deploy via CI; never edit DAGs/models directly in production UI | DevOps | 🟧 HIGH | Airflow DAG edited in web UI — 3 months later, git and deployed versions have diverged |
