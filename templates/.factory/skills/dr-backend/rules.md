# 📜 RULE REGISTRY — Backend & API Engineering

41 industry-specific decision rules, best practices, and anti-patterns.

| # | Rule | Category | Severity | Anti-Pattern (DO NOT) |
|---|---|---|---|---|
| R001 | All public-facing entity IDs must be UUIDv7 or ULID — never sequential integers, never UUIDv4 for high-insert tables (index fragmentation) | Data Modeling | 🟥 CRITICAL | `id SERIAL PRIMARY KEY` exposed in API responses |
| R002 | Every foreign key column must have a corresponding index — verify with `EXPLAIN SELECT … WHERE fk_col = ?` showing `Index Scan` not `Seq Scan` | Performance | 🟥 CRITICAL | Adding FK constraint without analyzing query plan on joined tables |
| R003 | Every mutating endpoint (POST/PUT/PATCH/DELETE) must accept `Idempotency-Key` header, store key→response mapping for ≥24 hours, return stored response on replay | Resilience | 🟥 CRITICAL | Relying on DB unique constraints alone for idempotency; `UNIQUE` violations leak internal state |
| R004 | All errors must exit through a single, centralized error handler that produces a consistent JSON envelope: `{"error":{"code":"STRING_CODE","message":"human","request_id":"uuid","timestamp":"ISO8601"}}` — never raw stack traces in production | Error Handling | 🟥 CRITICAL | `throw new Error("DB connection failed")` caught by Express default handler |
| R005 | Database migrations must be declarative and reversible — every `up()` must have a corresponding `down()` that restores the prior schema without data loss; test rollback in CI | Database | 🟥 CRITICAL | `DROP COLUMN` in `up()` with no `down()`; data-destructive migrations without explicit `--force` flag |
| R006 | All external service calls (HTTP, gRPC, message brokers) must be wrapped in a circuit breaker with configurable thresholds: failure-count ≥5 in 60s window → OPEN for 30s → HALF_OPEN → CLOSED | Resilience | 🟥 CRITICAL | Unbounded retries against a degraded downstream; no fallback response defined |
| R007 | Connection pools must be explicitly sized: `max = (CPU_cores * 2) + effective_spindle_count` for PostgreSQL; `max = CPU_cores * 4` for Redis; never rely on library defaults blindly | Performance | 🟧 HIGH | `pool: { max: 10 }` in production without calculating based on instance size |
| R008 | All list/query endpoints must enforce pagination with a server-side `maxPageSize` (default 100, max 1000); cursor-based pagination preferred over offset-based for datasets >10K rows | API Design | 🟧 HIGH | `SELECT * FROM orders` with no LIMIT; `?page=1&size=999999` accepted |
| R009 | Rate limiting must use GCRA (Generic Cell Rate Algorithm) or sliding-window-log — never fixed-window (thundering herd at boundary); emit `RateLimit-Limit`, `RateLimit-Remaining`, `RateLimit-Reset` headers on every response | API Design | 🟧 HIGH | Token bucket without burst allowance; missing rate-limit headers on non-limited endpoints |
| R010 | Every write to the primary database that has a side-effect (email, push, webhook, analytics event) must use the Transactional Outbox pattern — write event to `outbox` table in same DB transaction, separate worker polls and publishes | Data Consistency | 🟥 CRITICAL | `db.save(order); emailService.send(order);` — if email fails, DB is committed |
| R011 | Environment variables must never be hardcoded — read via a typed config layer that validates at startup (envalid, convict, Pydantic Settings); unknown keys must cause startup failure, not silent ignore | Configuration | 🟧 HIGH | `process.env.DATABASE_URL` scattered across 47 files with no validation |
| R012 | All SQL queries must use parameterized/bound statements — never string interpolation or concatenation of user input; scan codebase with regex `(execute|query)\(.+\\$\\{.+\}|.+\+.+\)` in JS/TS | Security | 🟥 CRITICAL | `` db.query(`SELECT * FROM users WHERE id = ${req.params.id}`) `` |
| R013 | Sensitive data (PII, secrets, tokens) must never appear in log output — implement a redaction layer that scrubs fields matching patterns like `password`, `token`, `secret`, `ssn`, `credit_card`, `authorization` before log emission | Security | 🟥 CRITICAL | `logger.info({ body: req.body })` on login endpoint |
| R014 | Long-running operations (>2s) must be executed asynchronously with a job queue — return `202 Accepted` with a `Location` header pointing to a status endpoint; client polls or receives webhook callback | API Design | 🟧 HIGH | Holding an HTTP connection open for 45 seconds waiting for video transcoding |
| R015 | Every API must have a health-check endpoint (`GET /health`) that returns 200 only when all critical dependencies (DB, Redis, message broker) pass their connectivity checks — include per-dependency status and latency | Observability | 🟨 MEDIUM | `/health` returning `{"status":"ok"}` without actually pinging the database |
| R016 | N+1 query detection is mandatory — if a loop body contains a DB call, you must refactor to a batched query, JOIN, or DataLoader; use an automated detector (e.g., `sequelize-bouncer`, Django nplusone) | Performance | 🟧 HIGH | `for (const user of users) { user.posts = await db.posts.findByUserId(user.id) }` |
| R017 | All datetime fields must be stored and transmitted in UTC — API inputs parsed as ISO 8601 with offset, stored as `TIMESTAMPTZ` (PostgreSQL) or equivalent; API outputs in UTC ISO 8601 with `Z` suffix | Data Integrity | 🟧 HIGH | `new Date().toLocaleString()` stored in DB; `TIMESTAMP WITHOUT TIME ZONE` in PostgreSQL |
| R018 | Read-after-write consistency: any endpoint that creates a resource and returns it must read from the primary (writer) node, not a read replica — replica lag causes 404 on subsequent GET of just-created resource | Data Consistency | 🟧 HIGH | `POST /orders → 201 { id: "abc" }` … `GET /orders/abc → 404` due to replication lag |
| R019 | Every API response that includes a collection must return a consistent envelope: `{"data":[…],"pagination":{"cursor":"...","hasMore":true},"meta":{"total":1234}}` — never bare arrays at the top level | API Design | 🟨 MEDIUM | `[{"id":1},{"id":2}]` as a top-level JSON array |
| R020 | All database queries must have a statement timeout — set `statement_timeout = '30s'` (PostgreSQL) or `max_execution_time = 30000` (MySQL) at session level for OLTP queries; analytics queries use a separate connection with higher timeout | Performance | 🟧 HIGH | Query running for 12 minutes holding a connection from a pool of 20 |
| R021 | Use composite unique constraints for business keys, not just surrogate PKs — `UNIQUE(tenant_id, email)` prevents race conditions at DB level; application-level uniqueness checks have TOCTOU gaps | Data Integrity | 🟧 HIGH | `const exists = await db.findByEmail(email); if (!exists) await db.insert(...)` with no UNIQUE constraint |
| R022 | Soft deletes must use `deleted_at TIMESTAMPTZ NULL` with a partial index `WHERE deleted_at IS NULL` — never `is_deleted BOOLEAN DEFAULT FALSE` (cannot index effectively, no timestamp for TTL purging) | Data Modeling | 🟨 MEDIUM | `is_deleted BOOLEAN NOT NULL DEFAULT 0` with no partial index |
| R023 | Batch operations must have a configurable max batch size (default 1000) — never unbounded; return `400 Bad Request` for oversized batches with clear error message stating the limit | API Design | 🟧 HIGH | `POST /bulk-delete { ids: [ … 50,000 IDs … ] }` with no limit |
| R024 | All inter-service communication must carry a trace context (`traceparent` header per W3C Trace Context) — propagate to all downstream calls; generate new span for each logical operation | Observability | 🟧 HIGH | Calling downstream service without forwarding `traceparent` — breaks distributed tracing |
| R025 | Graceful shutdown: on SIGTERM/SIGINT, stop accepting new requests, finish in-flight requests (with timeout of 30s), close DB connections, close message broker connections, flush logs, then exit — Kubernetes sends SIGTERM before SIGKILL | Resilience | 🟧 HIGH | `process.exit(0)` on SIGTERM — drops in-flight requests |
| R026 | Cache keys must be namespaced and versioned: `v2:service:entity:{id}:hash` — version prefix enables cache-busting on schema changes without key collisions | Caching | 🟨 MEDIUM | `cache.set(`user:${id}`, data)` — no version, no namespace, collision-prone |
| R027 | All file upload endpoints must validate MIME type by magic bytes (file signature), not by file extension or `Content-Type` header — use `file-type` or `libmagic` bindings | Security | 🟥 CRITICAL | `if (file.mimetype === 'image/png')` without checking first 8 bytes of file |
| R028 | Every consumer of a message queue/topic must handle messages idempotently — use message ID as dedup key, checkpoint after processing, tolerate at-least-once delivery | Messaging | 🟧 HIGH | Assuming exactly-once delivery from Kafka without consumer-side dedup |
| R029 | Connection strings and secrets must never be in source code, config files, or Dockerfiles — use a secrets manager (Vault, AWS Secrets Manager, Doppler) or at minimum env vars injected at runtime; scan commits with `gitleaks` or `truffleHog` in CI | Security | 🟥 CRITICAL | `const DB_PASSWORD = "super_secret_123"` committed to repository |
| R030 | Database index strategy: create indexes that match query patterns (WHERE, JOIN, ORDER BY, GROUP BY columns in that priority); drop unused indexes (monitor `pg_stat_user_indexes.idx_scan = 0` over 30 days) — indexes slow writes | Performance | 🟨 MEDIUM | Adding index on every column "just in case"; never reviewing index usage statistics |
| R031 | Event schemas must be backward-compatible by default — new fields must have defaults; never change field types; use schema registry (Confluent, Apicurio) with compatibility mode FULL_TRANSITIVE | Data Contracts | 🟧 HIGH | Changing `user_id` from string to integer in published event v2 — breaks all consumers |
| R032 | Retry logic must use exponential backoff with jitter: `delay = min(cap, base * 2^attempt) + random(0, delay * 0.1)` where base=100ms, cap=60s, max_attempts=5 | Resilience | 🟧 HIGH | Retrying every 2 seconds exactly — thundering herd in distributed systems |
| R033 | Before inserting/updating, validate data against the schema constraints that exist in the DB — application-level validation catches 95% of errors before hitting the DB and producing opaque constraint-violation errors | Input Validation | 🟧 HIGH | Relying solely on DB CHECK constraints and catching generic `QueryFailedError` |
| R034 | All monetary values must be stored as integers in the smallest currency unit (cents, satoshis) — never FLOAT/DOUBLE; use NUMERIC(19,4) for precise fractional amounts when needed (e.g., exchange rates, tax calculations) | Data Integrity | 🟥 CRITICAL | `price DECIMAL(10,2)` where 0.1 + 0.2 might equal 0.30000000000000004 |
| R035 | Multi-tenant applications must enforce tenant isolation at the data-access layer — every query must include `WHERE tenant_id = $current_tenant`; use Row-Level Security (RLS) in PostgreSQL as a defense-in-depth measure | Security | 🟥 CRITICAL | Relying on application code alone to filter by tenant — one missing WHERE clause leaks data |
| R036 | Log levels must be used correctly: ERROR (requires human action), WARN (system degraded but self-healing), INFO (business events: order placed, user signed up), DEBUG (developer diagnostics), TRACE (full request/response dump — never in production) | Observability | 🟨 MEDIUM | `logger.error("User not found")` on a normal 404 — generates alert fatigue |
| R037 | Database transactions must be as short as possible — no external API calls, no file I/O, no user-input waiting inside a transaction boundary; the maximum transaction duration should be <500ms for OLTP workloads | Performance | 🟧 HIGH | Opening a transaction, calling Stripe API, sending email, then committing — holds DB locks for seconds |
| R038 | All dependency injection must be explicit — pass dependencies through constructor/factory functions, never use service-locator anti-pattern (global `Container.get()` in every file); enables testing with mocks/stubs | Architecture | 🟨 MEDIUM | `const db = require('../../db').default` at the top of every service file — untestable |
| R039 | GraphQL APIs must define `maxDepth`, `maxComplexity`, and query cost analysis to prevent DoS via deeply nested or exponentially expensive queries (e.g., `{ users { posts { comments { user { posts { … }}}}}}`) | API Design | 🟧 HIGH | Exposing GraphQL endpoint with no query depth or complexity limits |
| R040 | Metrics must follow the RED method for every endpoint: Rate (requests/sec), Errors (failed requests/sec), Duration (p50/p90/p99 latency) — minimum; add Utilization (connection pool saturation, queue depth) for infrastructure-critical services | Observability | 🟨 MEDIUM | Only tracking CPU/memory — no per-endpoint error rate or latency breakdown |
| R041 | API versioning: URL path versioning (`/v1/`, `/v2/`) for breaking changes; header-based versioning (`Accept: application/vnd.api+json; version=2`) for minor variations; never mix both strategies on the same resource | API Design | 🟨 MEDIUM | `/api/getOrderV2`, `/api/get_order_new`, `/api/v1.5/orders` — three different versioning schemes |
