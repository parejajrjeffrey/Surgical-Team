# 📜 RULE REGISTRY — Security & Compliance Engineering

37 security-specific decision rules, best practices, and anti-patterns.

| # | Rule | Category | Severity | Anti-Pattern (DO NOT) |
|---|---|---|---|---|
| R001 | Never accept a CRITICAL or HIGH vulnerability without a time-bound risk acceptance — CRITICAL→24h, HIGH→7d; risk acceptance must have compensating control, justification, expiration date, and approver | Vulnerability Mgmt | 🟥 CRITICAL | `// TODO: fix later` on a critical RCE CVE — 18 months later still there |
| R002 | All IAM policies must follow least privilege — no `"Resource": "*"` with `"Action": "*"`; enumerate specific actions; use conditions for boundary enforcement; run IAM Access Analyzer | IAM | 🟥 CRITICAL | `AdministratorAccess` attached to every Lambda, EC2, and developer |
| R003 | All secrets must be stored in a secrets manager (Vault, AWS Secrets Manager), never in code, config files, or env files committed to git; pre-commit hooks + CI scans must block secrets | Secret Mgmt | 🟥 CRITICAL | `API_KEY=sk-proj-abcd1234` in `.env` that's `git add`-ed |
| R004 | Multi-factor authentication must be enforced for all human users and privileged operations — console, API, IAM changes, billing; service accounts use OIDC federation, not long-lived keys | IAM | 🟥 CRITICAL | Root AWS account with no MFA, access key from 2021 used daily |
| R005 | Encryption at rest (AES-256-GCM) for all data stores and in transit (TLS 1.3, mTLS); KMS keys must have automatic rotation enabled | Data Security | 🟥 CRITICAL | S3 bucket with 2M PII records, encryption disabled, public-read ACL, HTTP |
| R006 | SQL Injection prevention: every query must use parameterized/bound parameters — never string concatenation, template literals, or `format()` with user input; raw queries require security review | AppSec | 🟥 CRITICAL | `db.execute(f"SELECT * FROM users WHERE email = '{input}'")` |
| R007 | XSS prevention: all user input in HTML must be context-encoded; `dangerouslySetInnerHTML`/`v-html` only with DOMPurify sanitization | AppSec | 🟥 CRITICAL | `<div>{user.bio}</div>` where bio contains `<img src=x onerror=...>` |
| R008 | Security groups must never expose sensitive ports (22, 3389, 3306, 5432, 6379, 27017) to 0.0.0.0/0 — use bastion, SSM Session Manager, or VPN | Network | 🟥 CRITICAL | SG with `ingress { from_port=22, cidr_blocks=["0.0.0.0/0"] }` |
| R009 | Logging must capture all security events: auth success/failure, authZ decisions, data access, config changes, privilege escalations; logs immutable (WORM), shipped to SIEM | Logging | 🟥 CRITICAL | No CloudTrail in us-east-1 — attacker compromises undetected |
| R010 | Dependencies must be continuously scanned — CI pipeline runs `npm audit`/`pip-audit`/`trivy` on every PR; new CRITICAL/HIGH CVEs block merge | Vulnerability Mgmt | 🟧 HIGH | `package-lock.json` with 47 known vulns, 12 CRITICAL, last updated 14 months ago |
| R011 | Auth tokens must be short-lived — JWT ≤1h access + ≤24h refresh with rotation; API keys rotated ≤90d; OAuth refresh rotation prevents replay; session ≤15min idle, ≤8h absolute | Auth | 🟧 HIGH | JWT with `exp: 9999999999` — stolen once, valid forever |
| R012 | Public S3/GCS/Azure Blobs must be blocked by default — Block Public Access at account level; no public-read ACL; use CloudFront/OIDC for authenticated delivery | Cloud Security | 🟥 CRITICAL | `aws_s3_bucket_public_access_block` not set — dev enables public-read, forgets |
| R013 | CSRF protection: all state-changing requests require CSRF token or custom header (SPA with SameSite); CORS whitelist specific origins, never `*` with credentials | AppSec | 🟧 HIGH | `Access-Control-Allow-Origin: *` + `Access-Control-Allow-Credentials: true` |
| R014 | Security headers on all HTTP responses: HSTS(max-age=31536000;includeSubDomains), X-Content-Type-Options:nosniff, X-Frame-Options:DENY, strict CSP (no unsafe-inline/eval), Referrer-Policy, Permissions-Policy | AppSec | 🟧 HIGH | Production website with 0 security headers |
| R015 | Root account never used for daily operations — individual IAM users/roles; root MFA hardware key; root credentials stored secure offline with break-glass procedure | IAM | 🟥 CRITICAL | Root access key used in CI/CD "because it's easier than configuring roles" |
| R016 | Data retention and deletion policies defined and automated — PII/PHI deleted after purpose + legal hold; lifecycle policies for logs; GDPR right to erasure | Data Privacy | 🟧 HIGH | Customer requests deletion per GDPR — company forgets 47 backup snapshots and analytics tables |
| R017 | Network defense in depth — separate VPCs prod/non-prod, private subnets for data, NAT Gateway, NACLs+SGs, WAF on public endpoints, DDoS protection, VPC Flow Logs | Network | 🟧 HIGH | Single VPC, everything in public subnets, no WAF, no NACLs |
| R018 | IR plan documented, tested, accessible offline — team contacts, escalation path, communication templates, forensic procedures; tabletop quarterly, live-fire annually | IR | 🟥 CRITICAL | No IR plan — CTO learns about breach from customer tweet 6 hours later |
| R019 | Input validation server-side always, client-side optionally — never trust client-side validation; validate type, length, format, range, allowed chars (allowlist); reject invalid | AppSec | 🟥 CRITICAL | `<input maxlength="50">` — attacker sends 50MB payload via curl |
| R020 | File uploads secured — validate MIME by magic bytes (not extension), enforce max size, scan for malware, store outside webroot, serve via signed URLs, rename to random UUIDs | AppSec | 🟥 CRITICAL | `mv $_FILES['file']['tmp_name'] "/uploads/".$_FILES['file']['name']` — path traversal to RCE |
| R021 | Container images minimal and scanned — distroless/base, remove shells/compilers, non-root, read-only rootfs, Trivy→0 CRITICAL, cosign signed, SLSA attested | Container | 🟧 HIGH | `FROM ubuntu:latest` runs as root, has curl/wget/python3/gcc |
| R022 | Password policy NIST 800-63B — min 12 chars, no complexity rules, check HaveIBeenPwned API (k-anonymity), bcrypt/argon2id ≥250ms, never SHA-256/MD5 | Auth | 🟧 HIGH | 8 chars, uppercase+lowercase+number+symbol, change every 30 days — users write on sticky notes |
| R023 | Privileged access JIT and JEA — no standing admin; users request elevation with ticket+TTL(max 4h); all privileged sessions logged and auditable; break-glass with alerting | IAM | 🟧 HIGH | 50 engineers with `AdministratorAccess` 24/7 — one compromised credential owns everything |
| R024 | Certificate management automated — cert-manager for K8s, ACM for AWS, Let's Encrypt with auto-renewal ≥30d; monitor expiry and alert at 30/14/7/3/1 days | PKI | 🟧 HIGH | Manual cert via `openssl` set to expire in 90 days — expires Saturday 2am |
| R025 | Data classification implemented and enforced — Public/Internal/Confidential/Restricted; PII/PHI/PCI is Restricted; controls based on classification; never mix levels in same store | Data Security | 🟧 HIGH | PII, PCI card numbers, and public blog posts in same DB, same permissions |
| R026 | All security findings tracked to closure — vuln→ticket→fix→verify→close; weekly security review; SLAs based on severity; MTTR and recurrence tracked | Vulnerability Mgmt | 🟧 HIGH | Pentest finds 23 vulns in January — PDF in Slack, 8 still unfixed in October |
| R027 | Software supply chain secured — lockfiles committed, pinned versions (exact, no ^/~), private registry, SRI hashes for CDN, SLSA Level 3+ attestation, signed commits | Supply Chain | 🟧 HIGH | `"left-pad": "^1.0.0"` — unpublished breaks CI; or attacker publishes 1.0.1 with malware |
| R028 | Remote access zero-trust — no long-lived VPN creds; SSO+MFA+device posture; micro-segmentation; session recording for privileged access; access reviews quarterly | Network | 🟧 HIGH | VPN with shared creds, no MFA, full network access once connected |
| R029 | Sensitive data masked/tokenized in non-production — no production PII in staging/QA/dev; synthetic data or scrub; scan for PII in non-prod automatically | Data Privacy | 🟧 HIGH | Production DB restored to staging weekly — customer table with real PII available to every dev |
| R030 | API rate limiting to prevent abuse — per-endpoint per-consumer; GCRA or sliding window; auth endpoints heavily limited (5/IP/min with backoff); GraphQL cost analysis | AppSec | 🟧 HIGH | POST /login with no rate limit — brute-force 10K passwords in 5 minutes |
| R031 | Security control changes need peer review — IAM, SGs, encryption, auth middleware all require second approval; no self-approval; CODEOWNERS for security paths | Change Mgmt | 🟧 HIGH | Developer opens port 22 to 0.0.0.0/0 to debug, self-approves PR, forgets to revert |
| R032 | Threat modeling for new features and significant changes — STRIDE methodology during design review, not post-implementation checkbox | Threat Modeling | 🟨 MEDIUM | New payment feature — no threat model, price in POST body modifiable → $0.01 purchases |
| R033 | Personal devices/BYOD governed — MDM enrollment, containerized apps, remote wipe, minimum OS version, no corporate data on unmanaged devices | Endpoint | 🟨 MEDIUM | CEO iPad no passcode, no MDM, email synced — left in Uber, finder has 10 years of email |
| R034 | Security awareness ongoing, not annual — phishing simulations monthly; targeted training for clickers; OWASP Top 10 for devs annually; secure coding guidelines; new hire onboarding week 1 | Culture | 🟨 MEDIUM | Annual 30-min security video at 2x speed — 40% still click phishing links monthly |
| R035 | API auth never relies on obscurity — no "secret" endpoints without auth; no hardcoded tokens in mobile apps/SPAs; use OAuth 2.1 with PKCE for SPAs/mobile; server-to-server mTLS or signed JWTs | AppSec | 🟥 CRITICAL | `/api/internal/admin-panel` with no auth — "undocumented so no one will find it" — found via JS source map |
| R036 | Backup integrity and restoration tested — backups not "done" until restored+verified; quarterly restore test; immutable (WORM) backups; multi-region copies; offline/air-gapped critical copy | DR | 🟧 HIGH | Nightly backups to same server, never tested restore — ransomware encrypts DB AND backup files |
| R037 | All security exceptions must have finite lifespan — risk acceptance expires ≤12mo (typically 90d); must re-approve; automated reminders; escalation if unremediated | Governance | 🟧 HIGH | "Temporary" exception from 2022 — still open in 2026, 4 devs who knew about it left the company |
