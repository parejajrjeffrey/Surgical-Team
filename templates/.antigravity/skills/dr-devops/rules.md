# 📜 RULE REGISTRY — DevOps & Infrastructure

39 infrastructure-specific decision rules, best practices, and anti-patterns.

| # | Rule | Category | Severity | Anti-Pattern (DO NOT) |
|---|---|---|---|---|
| R001 | All container images must be referenced by SHA256 digest, never by mutable tag (`:latest`, `:3`, `:develop`) — digests guarantee immutability and auditability across all environments | Container Security | 🟥 CRITICAL | `image: nginx:latest` or `image: myapp:3` in Kubernetes manifests |
| R002 | Terraform state must be stored in a remote backend with locking enabled (S3 + DynamoDB, GCS, Azure Blob, Terraform Cloud) — never commit `.tfstate` to git; `.gitignore` must contain `*.tfstate*` and `.terraform/` | IaC | 🟥 CRITICAL | `git add terraform.tfstate`; using `local` backend for team projects |
| R003 | Every Kubernetes Deployment must define resource requests AND limits for every container — requests = limits for Guaranteed QoS in production; at minimum requests must be set; no "unbounded" pods | Kubernetes | 🟥 CRITICAL | Deployments with empty `resources: {}` — OOM kills, noisy neighbor, unbounded scaling |
| R004 | All containers must run as non-root (runAsNonRoot: true, runAsUser ≥ 10000) with readOnlyRootFilesystem: true, allowPrivilegeEscalation: false, and capabilities.drop: [ALL] — add only required capabilities with justification | Container Security | 🟥 CRITICAL | Container running as UID 0 with full capabilities — one RCE = cluster compromise |
| R005 | NetworkPolicy must default-deny all ingress and egress in every namespace — add explicit allow-rules only for validated communication paths (app→DB:5432, monitoring→all:9090); no allow-all-to-all rules | Kubernetes | 🟥 CRITICAL | No NetworkPolicy defined — any pod can reach any other pod on any port |
| R006 | All secrets must be stored in a dedicated secrets manager (Vault, AWS Secrets Manager, External Secrets Operator), never in Kubernetes Secrets without encryption at rest — enable etcd encryption or use Sealed Secrets/ESO | Secret Management | 🟥 CRITICAL | `kubectl create secret generic db-pass --from-literal=password=hunter2` committed to git |
| R007 | CI/CD pipelines must include mandatory security scanning gates: container vulnerability scan (Trivy/Grype → 0 CRITICAL/0 HIGH), IaC scan (checkov/tfsec → 0 FAILED), secret scan (gitleaks/truffleHog → 0 findings) — block merge on failure | Pipeline Security | 🟥 CRITICAL | Pipeline that only runs `npm test` and deploys — no security scanning |
| R008 | Production deployments must use canary or blue-green strategy with automated metric verification — never in-place rolling update without traffic splitting; automated rollback on error-rate or latency degradation > threshold | Deployment Strategy | 🟧 HIGH | `kubectl apply -f deployment.yaml` directly to production with no canary |
| R009 | Every alert must have a severity level (P1-P5), a runbook link in its annotations, and a defined SLO-based threshold — never alert on symptoms without a corresponding runbook; P1 alerts route to on-call, P5 are dashboards only | Observability | 🟧 HIGH | `alert: HighCPU` with no runbook, no severity — someone gets paged at 3am, has no idea what to do |
| R010 | Infrastructure must be fully reproducible from code alone — quarterly fire-drill: `terraform destroy` in sandbox account, `terraform apply`, verify system health; no manual cloud console changes tolerated | Disaster Recovery | 🟧 HIGH | "That EC2 instance was created manually 2 years ago, no one knows what it does" |
| R011 | All cloud resources must use a consistent tagging strategy: `Environment`, `Service`, `Team`, `CostCenter`, `ManagedBy` (Terraform/helm/argo) — block creation of untagged resources via AWS SCP or Azure Policy | Cost Management | 🟧 HIGH | Resources with 0 tags — untraceable cost, unknown ownership |
| R012 | Database backups must be automated, encrypted, and tested — restore from latest backup in staging at least quarterly; verify RPO ≤ 1hr, RTO ≤ 4hr for critical workloads; document recovery time in runbook | Disaster Recovery | 🟥 CRITICAL | Assuming RDS automated backups work without ever testing a restore |
| R013 | TLS must be enforced everywhere — minimum TLS 1.3 for all ingress; mTLS between all mesh-connected services (Istio STRICT mode); certificates auto-renewed via cert-manager with ≥30-day renewal window | Security | 🟥 CRITICAL | HTTP endpoint exposed to internet; TLS terminated at LB, plain-text between LB and pod |
| R014 | PodDisruptionBudget must be set for every Deployment with ≥2 replicas — `maxUnavailable: floor(N*0.25)` or `minAvailable: ceil(N*0.75)`; prevents all pods being evicted simultaneously during node drain | Kubernetes | 🟧 HIGH | 3-replica Deployment with no PDB — cluster autoscaler drains all 3 pods at once → outage |
| R015 | Log retention must be explicitly configured — minimum 30 days for application logs, 90 days for audit/security logs, 1 year for compliance logs; logs must be shipped off-cluster (Loki, ELK, CloudWatch) not stored on ephemeral node disks | Observability | 🟧 HIGH | Relying on `kubectl logs` for debugging a production issue from 2 weeks ago |
| R016 | Docker images must use multi-stage builds — final image must not contain build tools (gcc, make, git), package manager caches, dev dependencies, or source code beyond compiled artifacts; target distroless or minimal base (alpine, chainguard) | Container Security | 🟧 HIGH | Single-stage Dockerfile with `node:latest` base, `node_modules/` with 2GB of devDependencies |
| R017 | All Helm charts must be versioned (semver) and published to an OCI-compliant registry — never deploy from a git branch reference; `helm install` must pin chart version explicitly, not `latest` or floating `^1.0.0` | Kubernetes | 🟧 HIGH | `helm install myapp ./chart` from a developer's laptop, chart version 0.1.0 from 18 months ago |
| R018 | IAM roles and service accounts must follow least-privilege — use IRSA (IAM Roles for Service Accounts) in EKS, Workload Identity in GKE, or pod identity in AKS; never embed long-lived IAM credentials in pods; audit with IAM Access Analyzer | Security | 🟥 CRITICAL | `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY` in a Kubernetes Secret for a service account with `AdministratorAccess` |
| R019 | Kubernetes probes must be correctly configured: liveness (checks restart-worthy deadlocks, initialDelaySeconds=p50_startup+30s), readiness (checks if accepting traffic, initialDelaySeconds=p50_startup+10s), startup (replaces liveness during slow boot); never use the same endpoint for liveness and readiness | Kubernetes | 🟧 HIGH | Liveness probe checking DB connection with `initialDelaySeconds: 5` — restarts pod during DB migration |
| R020 | All Terraform modules must pin provider versions with pessimistic constraint operator `~>` (e.g., `~> 5.0`) and lock provider hashes in `.terraform.lock.hcl` committed to git — prevents unexpected provider upgrades | IaC | 🟧 HIGH | `aws = { source = "hashicorp/aws" }` with no version constraint — `terraform init` pulls latest major version, breaks everything |
| R021 | CI/CD pipeline credentials must use OIDC federation, not long-lived API tokens — GitHub Actions → AWS via `aws-actions/configure-aws-credentials` with OIDC; GitLab CI → cloud via JWT; never store cloud credentials as repository secrets | Pipeline Security | 🟥 CRITICAL | `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY` copied into GitHub repository secrets from a root IAM user |
| R022 | Cluster autoscaling must be tested — schedule a pod that requests > available capacity, verify new node joins within 5 minutes; set `scale-down-unneeded-time` to ≥10 min to prevent aggressive scale-down thrashing; use Karpenter for faster, bin-packing-optimized scaling | Kubernetes | 🟨 MEDIUM | Static node count — paying for idle capacity 24/7 |
| R023 | All Ingress/Gateway resources must configure appropriate timeouts — `proxy-read-timeout`, `proxy-send-timeout`, `proxy-body-size` must match application expectations; default values (30s read timeout, 1MB body) often cause 504/413 errors in production | Networking | 🟧 HIGH | Default NGINX Ingress timeout of 60s — long-running API calls consistently timeout |
| R024 | Service meshes (Istio/Linkerd) must enforce mTLS STRICT mode — `PeerAuthentication` policy at mesh or namespace level; no PERMISSIVE mode in production; AuthorizationPolicy for service-to-service access control beyond just mTLS | Security | 🟧 HIGH | Installing Istio, enabling PERMISSIVE mTLS, never tightening to STRICT — plaintext traffic silently accepted |
| R025 | Infrastructure cost must be estimated in CI — run Infracost in pull requests to show `$ diff vs main` for Terraform changes; block merge if cost increase exceeds agreed threshold without explicit approval | Cost Management | 🟨 MEDIUM | Adding 50 `db.r6g.16xlarge` instances with no cost visibility until AWS bill arrives |
| R026 | Graceful termination must be configured — `terminationGracePeriodSeconds` must exceed the time needed to drain in-flight requests + close DB connections + flush logs; `preStop` hook must trigger draining | Kubernetes | 🟧 HIGH | `terminationGracePeriodSeconds: 5` — SIGKILL sent 5 seconds after SIGTERM, drops all in-flight requests |
| R027 | Container image registries must be private with authentication required — never pull from public registries in production (supply chain risk); use imagePullSecrets or ECR/GCR/ACR native integration; regularly audit pull-through cache for stale images | Container Security | 🟧 HIGH | `image: mycompany/backend:prod` from Docker Hub public repository — anyone could push a malicious image to that tag |
| R028 | Logging must use structured format (JSON) and stdout/stderr only (never files inside container) — container orchestrator captures stdout; use log shipper (Fluent Bit, Vector) to forward to aggregation | Observability | 🟨 MEDIUM | `console.log("User " + user.id + " logged in")` — unstructured, un-indexed, unsearchable |
| R029 | All Terraform changes must go through plan-review-apply cycle — `terraform plan` output posted as PR comment; at least one approver reviews plan before `terraform apply`; no `--auto-approve` flag in CI (except for sandbox/transient environments) | IaC | 🟧 HIGH | Developer runs `terraform apply --auto-approve` from local machine against production |
| R030 | PersistentVolumes must use a StorageClass with `reclaimPolicy: Retain` or snapshot-based backup before deletion — `Delete` reclaim policy only for truly ephemeral data (cache, temp); verify backups exist before PVC deletion operations | Kubernetes | 🟥 CRITICAL | `kubectl delete pvc data-mysql-0` — all production data instantly gone |
| R031 | Node pools must be segregated by workload type — system-critical (CoreDNS, metrics-server, ingress-controller) on dedicated system nodes; application workloads on app nodes; GPU/ARM workloads on specialized nodes; use taints/tolerations + nodeSelectors to enforce | Kubernetes | 🟧 HIGH | All workloads on a single node pool — a memory leak in a dev service OOMs CoreDNS, cluster DNS dead |
| R032 | Every service must have an SLO (Service Level Objective) defined and monitored — e.g., "99.9% of requests complete successfully and within 500ms over a 30-day rolling window"; error budgets calculated; SLO violations trigger P2 alerts; no SLO = no production deployment | Observability | 🟧 HIGH | "Our service has 99.999% uptime" — based on ICMP ping, not actual user-facing request success rate |
| R033 | Infrastructure documentation must live alongside code — `README.md` in every Terraform module, Helm chart, and infrastructure repo explaining: what it deploys, dependencies, input variables, output values, operational runbooks; never rely on tribal knowledge or outdated Confluence pages | Documentation | 🟨 MEDIUM | Terraform module with 47 variables, 0 documentation — only the original author knows what `var.bloop_threshold` does |
| R034 | Pod anti-affinity must be configured for HA — `podAntiAffinity.preferredDuringScheduling` with `topologyKey: kubernetes.io/hostname` for stateless services (≥3 replicas → spread across nodes); `requiredDuringScheduling` for stateful components that must not co-locate | Kubernetes | 🟧 HIGH | 3 replicas of critical service all scheduled on the same node — AZ failure = full outage |
| R035 | Infrastructure git repositories must enforce branch protection: require pull request, require ≥1 approval, require status checks to pass, require linear history, block force push to main; no exceptions for "emergency fixes" (use hotfix branch + PR) | Pipeline Governance | 🟧 HIGH | `main` branch with no protection — junior dev force-pushes at 4:55 PM on Friday, entire IaC repo destroyed |
| R036 | All external endpoints must be protected by a WAF (Web Application Firewall) — AWS WAF, Cloudflare WAF, or GCP Cloud Armor with OWASP Top 10 rules, rate-based rules, and IP reputation lists; WAF must be in blocking mode, not count-only, after tuning period | Security | 🟧 HIGH | ALB/NLB exposed directly to internet with no WAF — SQL injection, XSS, DDoS attacks unmitigated |
| R037 | Kubernetes API server audit logs must be enabled and shipped to a SIEM or security lake — capture: who did what, when, from which IP, with what result; retain for ≥90 days; alert on suspicious patterns | Security | 🟧 HIGH | No audit logging — if someone `kubectl exec`s into a production pod and dumps the database, there's no record |
| R038 | DNS must use split-horizon or private zones for internal services — internal service endpoints resolve to private IPs (VPC CIDR, ClusterIP); never expose internal hostnames in public DNS; use ExternalDNS for automatic Route53 record management | Networking | 🟨 MEDIUM | `db.internal.example.com` resolves to public IP — DB reachable from internet |
| R039 | All scheduled jobs (CronJobs, scheduled Lambda, Cloud Scheduler) must have: (1) deadline/timeout to prevent overlap, (2) concurrencyPolicy: Forbid for CronJobs, (3) idempotency guarantee, (4) failure alerting, (5) successful completion monitoring | Reliability | 🟧 HIGH | CronJob that sends invoices runs for 18 months, silently failing since month 3 — no one noticed |
