# 📜 RULE REGISTRY — QA & Testing Engineering

36 QA-specific decision rules, best practices, and anti-patterns.

| # | Rule | Category | Severity | Anti-Pattern (DO NOT) |
|---|---|---|---|---|
| R001 | Every bug fix MUST include a regression test that fails before the fix and passes after — the test must reproduce the exact conditions of the reported bug; no bug is "too trivial" for a regression test | Test Quality | 🟥 CRITICAL | Fixing a bug and closing the issue with no test — same bug will return in 6 months |
| R002 | Tests must be independently runnable — no test-ordering dependency, no shared mutable state between tests, every test creates its own data or uses isolated fixtures; use `--shard` or `--parallel` and verify no failures | Test Architecture | 🟥 CRITICAL | `test('B')` depends on database state created by `test('A')` — breaks in parallel, breaks when reordered |
| R003 | E2E tests must use semantic locators in priority order: `getByRole()` → `getByLabelText()` → `getByPlaceholderText()` → `getByText()` → `getByTestId()` — never CSS class names, never XPath, never deeply nested CSS selectors | E2E Testing | 🟥 CRITICAL | `page.locator('.col-md-4 > div:nth-child(3) > button.btn-primary-lg')` — breaks on any CSS refactor |
| R004 | API contract tests must validate response bodies against the OpenAPI JSON Schema, not just status codes — use `openapi-response-validator` or equivalent; a 200 with wrong-shaped body is a failure | API Testing | 🟥 CRITICAL | `expect(response.status).toBe(200)` with no body validation |
| R005 | Performance tests must assert on p95 and p99 latencies, not averages — averages hide tail-latency problems; set explicit k6 thresholds: `http_req_duration: ['p(95)<500', 'p(99)<1000']` | Performance Testing | 🟥 CRITICAL | Loading with 1000 VUs, reporting `avg: 200ms` while p99 is 8 seconds |
| R006 | All test data must be generated using factories/fixtures with randomized unique values per run — never hard-coded IDs, emails, or usernames that cause collisions between parallel test runs; use `faker` or `chance` libraries | Test Data | 🟧 HIGH | `const testUser = { email: 'test@test.com', id: 1 }` — parallel tests collide on unique constraint |
| R007 | Flaky tests (pass/fail intermittently without code change) must be quarantined immediately — move to a separate quarantine suite, create a ticket with `label:flaky-test`, resolve within 1 sprint; never ignore or comment out flaky tests | Test Quality | 🟥 CRITICAL | `test.skip('flaky')` with no follow-up; CI passes 60% of the time — team learns to ignore failures |
| R008 | Every test must have a single, clear reason to fail — if a test can fail for 5 different reasons, it's too broad; use Arrange-Act-Assert (AAA) pattern; one logical assertion per test | Test Design | 🟧 HIGH | `test('user flow')` with 47 assertions spanning login, profile, settings, logout — fails, no one knows why |
| R009 | Coverage thresholds must be enforced in CI — minimum 80% line coverage, 70% branch coverage, 80% function coverage; no lowering thresholds without architecture team approval | Code Coverage | 🟧 HIGH | `jest --coverageThreshold='{"global":{"lines":50}}'` — path of least resistance, coverage decays over time |
| R010 | Mutation testing must run periodically (weekly or per-release) — use StrykerMutator (JS/TS), pitest (Java), mutmut (Python), or go-mutesting; surviving mutants indicate tests that pass without verifying behavior | Test Quality | 🟨 MEDIUM | 95% code coverage, but mutation score of 20% — tests cover lines but don't verify actual outcomes |
| R011 | Accessibility tests must run as part of the E2E pipeline with axe-core or pa11y — enforce WCAG 2.2 AA minimum; block merge on critical or serious a11y violations | Accessibility | 🟧 HIGH | No a11y checks — dropdown menus unreachable by keyboard, modals without focus traps, 50 contrast violations |
| R012 | Test execution must be deterministic — no `setTimeout` waits, no `sleep(2000)`, no reliance on real-time clocks; use Playwright `waitFor*` assertions with timeouts, fake timers (jest.useFakeTimers), or explicit polling | E2E Testing | 🟧 HIGH | `await page.waitForTimeout(5000)` — passes on fast CI, fails on slow CI, wastes 5 seconds per test |
| R013 | All external dependencies in tests must be mocked or stubbed — no real HTTP calls, no real SMTP, no real payment gateways in unit/integration tests; use wiremock, nock, MSW, or test doubles; E2E tests may use sandbox/test-mode APIs | Test Isolation | 🟧 HIGH | Unit test sends a real email via SendGrid, hits Stripe API, and charges a credit card |
| R014 | Test failure output must be immediately actionable — screenshot (E2E), request/response dump (API), diff (unit assertion), trace file (Playwright); never just `expected true, received false` without context | Test Reporting | 🟧 HIGH | `AssertionError: expected [] to have length 1` — no indication what was expected or what went wrong |
| R015 | Idempotency must be tested for all mutating endpoints — send same request with same `Idempotency-Key` twice; verify: both return 2xx with identical body, only one side-effect occurred (check DB, check events emitted) | API Testing | 🟥 CRITICAL | Assuming idempotency works because middleware is attached — never actually tested |
| R016 | Rate limiting must be tested at both boundaries — send requests at exactly the limit (should pass), then one more (should get 429); verify `Retry-After` header is present and is a positive integer; verify limit resets after the window | API Testing | 🟧 HIGH | Testing rate limiting with only a few requests — never verifying the exact boundary or the reset |
| R017 | Visual regression tests must use a consistent viewport and deterministic rendering — disable animations, mock dates, mock random values, disable flaky images/maps; use pixelmatch threshold ≤0.1% | E2E Testing | 🟨 MEDIUM | Visual diff test on homepage with rotating testimonials, live Twitter feed, and an animated hero — fails every run |
| R018 | Test environments must be ephemeral and isolated — each test run gets a fresh schema/sandbox; no "shared staging DB" that accumulates state; use Testcontainers, Docker Compose, or cloud preview envs | Test Infrastructure | 🟧 HIGH | All developers and CI sharing one staging database — `test('delete all users')` runs, production-like data gone |
| R019 | Security tests must verify JWT/OAuth edge cases — tokens with `alg: "none"`, expired tokens (just past `exp`), tokens with tampered claims, tokens signed with wrong key, missing `iss`/`aud` validation | Security Testing | 🟥 CRITICAL | Only testing the happy-path of "valid token → 200" — never testing the attack surface |
| R020 | Test naming must follow convention: `should <expected behavior> when <condition>` or `MethodName_StateUnderTest_ExpectedBehavior` — tests must read like documentation | Test Design | 🟨 MEDIUM | `test('handles it')`, `test('works')`, `test('edge case 2')` — useless when reading CI failure logs |
| R021 | Browser console errors must be asserted in E2E tests — `page.on('console', ...)` → `expect(errors).toHaveLength(0)`; uncaught JS errors, 404s on critical assets, CORS violations all fail the test | E2E Testing | 🟧 HIGH | E2E test passes but browser console shows `Uncaught TypeError: Cannot read properties of undefined` |
| R022 | Load tests must include ramp-up period — start with 0 VUs, ramp gradually to target over ≥30 seconds; never start at max VUs (connection pool exhaustion, cold-start distortion); include ramp-down to detect resource cleanup issues | Performance Testing | 🟨 MEDIUM | `k6 run` with `stages: [{ target: 1000, duration: '0s' }]` — instant spike masks real performance profile |
| R023 | Test plan must be traceable to requirements — every test maps to a spec/AC/ticket via `@tag` or `tms` link; orphan tests (no requirements link) must be justified as regression catch or deleted as zombie tests | Test Documentation | 🟨 MEDIUM | 5000 tests in the suite, no one knows which are testing current behavior vs legacy cruft |
| R024 | Time-based logic must be tested with fake timers — `jest.useFakeTimers()`, `vi.useFakeTimers()`, or `freezegun`; tests that expire tokens, check schedules, or have TTLs must never actually `sleep()` | Unit Testing | 🟧 HIGH | `await new Promise(r => setTimeout(r, 61000))` to test a 60-second TTL — test takes 61 seconds |
| R025 | Database state must be reset between tests — truncate tables or rollback transaction; use `beforeEach` for setup and `afterEach` for cleanup; tests that leave side-effects in the database are the #1 cause of flaky test suites | Test Infrastructure | 🟥 CRITICAL | Test inserts `user@test.com`, next test inserts same email → UNIQUE constraint violation — flaky depending on execution order |
| R026 | Mobile responsiveness must be tested — set Playwright viewports for iPhone 15, iPad, and 1920p desktop; verify no horizontal scroll, tappable touch targets (≥44×44px per WCAG), readable font without zoom | E2E Testing | 🟧 HIGH | Only testing at 1920×1080 — hamburger menu broken on mobile, table overflows, users rage |
| R027 | Retry logic for flaky tests must be limited — max 2 retries in CI, never `retries: Infinity`; every retry must log a warning; if a test retries successfully in >5% of runs, it's a flaky test that must be fixed | Test Quality | 🟧 HIGH | `retries: 10` — test passes on the 7th attempt, everyone ignores the CI log spam |
| R028 | Snapshot tests must be reviewed like code — small, focused snapshots; use `toMatchInlineSnapshot()` for small outputs; never snapshot large objects without explicit intent and name | Unit Testing | 🟨 MEDIUM | `expect(fullPageHTML).toMatchSnapshot()` — 1500-line snapshot, no one reads diff, blindly updating on every change |
| R029 | API error responses must be tested for every defined error code in the OpenAPI spec — if spec says `404_RESOURCE_NOT_FOUND` and `409_VERSION_CONFLICT`, both must have tests; don't just test the 200 path | API Testing | 🟧 HIGH | Only testing happy-path 200 responses — 404, 409, 422 paths never exercised until a user hits them |
| R030 | Test data factories must produce valid-by-default data — `UserFactory.build()` must pass validation; override only the fields under test; prevents tests passing because of coincidence | Test Data | 🟧 HIGH | `const user = { name: '', email: 'not-an-email', age: -5 }` — test passes because validation is in a different layer |
| R031 | CI must fail fast on critical tests — run smoke tests (1-2 min) before full test suite (15-30 min); if smoke tests fail, skip full suite; saves CI minutes and developer waiting time | Test Infrastructure | 🟨 MEDIUM | Full test suite runs for 25 minutes, all tests fail because database connection string is wrong — every test times out |
| R032 | File upload endpoints must be tested with valid and malicious payloads — valid: correct MIME, within size limit; malicious: wrong MIME via magic bytes, oversized, empty file, SQL injection in filename, path traversal, polyglot files | Security Testing | 🟥 CRITICAL | Only testing file upload with one valid PNG and calling it done |
| R033 | Cross-browser testing must cover the supported matrix — if the app supports Chrome 90+, Firefox 90+, Safari 16+, then tests must run on all three; use Playwright's built-in browser channels | E2E Testing | 🟨 MEDIUM | "Works on my machine" — Chrome 127 on macOS, users on Firefox/Linux and Safari/iOS see a broken app |
| R034 | Test coverage reporting must differentiate meaningful from mechanical coverage — `/* istanbul ignore next */` allowed for truly untestable code; but every ignore comment must justify why; track coverage by critical path, not just global percentage | Code Coverage | 🟨 MEDIUM | 80% coverage achieved by testing getters, setters, and constructor assignments — core business logic at 0% |
| R035 | Performance test assertions must compare against a baseline — store historical performance data; flag regressions where p95 latency increases >20% or throughput drops >10% vs the 7-day rolling average | Performance Testing | 🟧 HIGH | Performance test passes because latency is under 500ms — last week it was 150ms, now it's 490ms, regression undetected |
| R036 | All test assertions must have descriptive failure messages — use the optional `message` parameter: `expect(value, 'User email should be normalized to lowercase').toEqual(expected)` | Test Quality | 🟨 MEDIUM | `expect(received).toBe(expected)` — failure output: `expected "admin" to be "ADMIN"` — no clue which field or why |
